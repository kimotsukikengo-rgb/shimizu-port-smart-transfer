1. データベース設計

設計方針

  • RDB（PostgreSQL推奨）を前提としたリレーショナル設計
  • 全テーブルに id（BIGSERIAL PK）、created_at、updated_at を共通付与
  • 外部キー制約でデータ整合性を保証
  • 論理削除（is_active / deleted_at）を基本方針とする（旅行業法上の帳票保管義務5年間）
  • JSON型は option_items、destinations 等の可変長配列データにのみ使用

2. 採番ロジック

受付番号（Reception Number）

形式: R-YYYYMMDD-NNN（例: R-20260326-001）

  • 寄港日の日付 + その日の連番（001から開始）
  • 事前予約（URL A）と当日即売（URL B）で同一連番体系を使用
  • 同時操作による重複を防ぐため、DB側でシーケンスまたは SELECT FOR UPDATE で排他制御
-- 推奨実装: DBシーケンス + アプリ側フォーマット
-- 日付ごとにカウンタをリセットする方式

-- 方法A: ordersテーブルのサブクエリで算出
SELECT CONCAT('R-', TO_CHAR(CURRENT_DATE, 'YYYYMMDD'), '-',
  LPAD((COUNT(*) + 1)::TEXT, 3, '0'))
FROM orders WHERE port_call_id = :port_call_id
FOR UPDATE;

-- 方法B: 専用カウンタテーブル
CREATE TABLE reception_counters (
  port_call_id BIGINT PRIMARY KEY REFERENCES port_calls(id),
  last_number  INTEGER DEFAULT 0
);
-- UPDATE ... SET last_number = last_number + 1 RETURNING last_number

W番号（Waiting Number）

形式: W-NNN（例: W-005）。寄港日ごとにリセット。物理カードの番号と1:1対応。

  • 寄港情報登録時に範囲（W-001〜W-060等）を事前設定
  • チャーターとワンウェイで共通連番（区分なし）
  • 車両単位で割り当て: 3台口 → W-005, W-006, W-007
  • スタッフが物理カード番号をシステムに手入力（バリデーション: 範囲内であること、未使用であること）

3. ステータス遷移設計

3.1 受注ステータス（orders.status）

受注完了→ 決済待ち→ 決済完了→ 配車待ち→ 帰着待ち→ 運行完了
遷移トリガーバリデーション副作用
受注完了 → 決済待ち受注登録完了（自動）reception_number が発行済み受付番号発行
決済待ち → 決済完了決済デスクで完了操作payment_method が入力済み、全車両にW番号割当済みdispatches レコード生成（vehicle_count分）
決済完了 → 配車待ち自動（決済完了と同時）
配車待ち → 帰着待ち配車担当が配車完了操作全dispatchesに driver_id, taxi_company_id 入力済み指示書PDF生成、メール/LINE送信
帰着待ち → 運行完了乗務員が完了報告送信全dispatchesの driver_report が submitteddispatch.status → 運行完了
任意 → キャンセルキャンセル操作cancel_reason が入力済みsales_amount を 0 に更新。W番号は解放（再利用不可、取消線表示）

画面表示ルール

管理画面・URL Bのステータスフィルタは「帰着待ち」「運行完了」の2値のみ表示。内部ステータスはAPIレスポンスで返すが、UIには簡略化して表示する。

内部ステータス画面表示
受注完了, 決済待ち, 決済完了, 配車待ち, 帰着待ち帰着待ち
運行完了運行完了
キャンセルキャンセル（取消線）

4. API設計

共通仕様

  • RESTful JSON API。レスポンスは { data, error, meta } 形式
  • 認証: URL A は不要。URL B / Admin は Bearer Token（JWT）
  • ページネーション: ?page=1&per_page=50（デフォルト50件）
  • エラー: HTTP status + { error: { code, message, details } }
  • 日時: ISO 8601 形式（YYYY-MM-DDTHH:mm:ssZ）

5. 画面コンポーネント設計

  • 4つのフロントエンドを共通コンポーネントライブラリで構築

6. 認証・認可設計

対象認証方式トークン有効期限アクセス範囲
URL A（顧客予約）認証不要（公開）—コース閲覧、予約フォーム、Stripe決済のみ
URL B（スタッフ）スタッフコード + パスワード → JWT12時間ログイン後、選択したロールの画面のみ。マスタGETは全ロール許可
Admin（管理画面）ID + パスワード → JWT12時間全API。CSV出力・マスタCRUDはadminロール限定
乗務員報告ワンタイムトークン（URL埋込み）当日23:59まで自分の dispatch_id に対する報告フォームのみ

ワンタイムトークン設計

配車確定時に driver_reports レコードを生成し、report_token（UUIDv4）を発行。

乗務員へのメールに https://{domain}/driver-report/{token} を記載。

トークンは token_expires_at（当日23:59）を超えると無効。提出済み（submitted_at IS NOT NULL）も無効。

9. エラーハンドリング方針

シナリオ対応方針
受付番号の競合DB排他制御（FOR UPDATE）でリトライ。最大3回。アプリ側で楽観ロック検知
W番号の重複入力UNIQUE制約で弾く。UIで「このW番号は使用済みです」メッセージ
Stripe決済失敗Stripe.jsでクライアント側エラー表示。サーバー側はWebhookで最終確認
PDF生成失敗再生成ボタンを提供。バックグラウンドジョブで非同期再試行
メール/LINE送信失敗送信ステータス（sent/failed）を記録。管理画面から再送信可能
オフライン時（URL B）PWA検討。最低限、受注入力画面のオフラインキャッシュを実装（オンライン復帰時に同期）
同時操作の競合楽観ロック（updated_at比較）。競合時はUI通知で最新データに再取得を促す
